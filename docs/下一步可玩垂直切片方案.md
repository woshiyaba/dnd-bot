# 下一步可玩垂直切片方案

> 目标：不要继续优先扩抽象、扩 agent、扩状态字段。下一阶段只做一件事：把「钟楼下的低语」跑成一个玩家能从网页打开、一路玩到结局的最小闭环。

## 结论

现在最应该做的功能是：**接通 `SessionEngine` 的真实 HTTP/WebSocket API，并用前端做出一局 10-15 分钟的可玩短冒险。**

当前项目最有价值的部分已经存在：

- `src/session/engine.py` 已经能驱动一整局冒险。
- `src/session/graph.py` 已经能把中央 DM、战斗子图和剧情推进串起来。
- `src/combat/` 已经能做可中断、可恢复、规则确定的战斗。
- `canon/whispers_bell_tower.json` 已经有一个短剧本骨架。

真正缺的不是更多底层能力，而是玩家体验闭环：

```text
玩家打开页面
  → 进入钟楼短剧本
  → 和 DM 对话
  → 做明检定并亲自掷骰
  → 触发战斗
  → 在战斗中选择行动和掷骰
  → 胜利或失败
  → 看到结局
```

这条链路跑通后，才能判断项目到底哪里好玩、哪里不好玩。现在继续做更复杂的 agent 或更多规则，短期内不如这个闭环有价值。

## 为什么这是第一优先级

D&D AI 项目真正好玩的点不是「AI 能聊天」，而是这五件事一起成立：

1. 玩家说自然语言，系统能转成世界行动。
2. 该掷骰时系统会停下来，让玩家参与。
3. 规则结算可靠，不像纯 LLM 那样乱编。
4. 战斗有状态、有风险、有结果。
5. DM 会把检定、战斗和剧情结果讲成连续故事。

这个项目已经选对了技术路线：**规则确定 + 叙事自由**。现在的风险不是地基不够，而是还没有一个可玩的切片证明这套地基能带来乐趣。

## 当前状态判断

### 已经有的

- `SessionEngine.start_session/message/submit/current_state` 已经定义了清晰的会话门面。
- 会话返回值已经归一成 `awaiting_input`、`interrupted`、`finished`。
- 中断 payload 已经包含 `directed_to.user_id`，可以支持前端路由「该谁掷什么」。
- 战斗子图已经通过 `run_combat` 嵌入会话主图，战斗结束后能写回世界状态。
- canon 系统已经能用 `campaign_id` 引用剧本，不需要把完整剧本塞进 checkpoint。

### 还没有的

- `src/app.py` 仍是 deepagents 模板入口，`POST /invoke` 没有调用 `SessionEngine`。
- 前端 `front/pc-dnd-bot/src/App.tsx` 还是 Vite starter，没有跑团界面。
- WebSocket 当前服务于模板流式事件，还没有定义 session 事件。
- `whispers_bell_tower` 作为完整可玩体验还没有经过浏览器端验证。

所以当前最短路径不是重写引擎，而是把现有引擎接到产品入口。

## MVP 范围

第一版只支持单人、单房间、单剧本。不要在这一版做多人、角色创建、背包全功能、地图编辑器或完整规则书。

### 必须有

- 启动「钟楼下的低语」剧本。
- 显示 DM 叙述和玩家消息。
- 玩家输入自然语言行动。
- 出现明检定时，前端显示掷 d20 入口。
- 出现战斗行动声明时，前端显示可选攻击、移动、放弃等合法行动。
- 显示玩家和敌人的 HP、当前状态、战斗结果。
- 战斗结束后继续回到 DM 叙述。
- 到达胜利或失败结局后，页面明确显示结局。

### 暂时不做

- 多人房间和旁观广播。
- 复杂角色创建器。
- 完整 D&D 法术系统。
- 大地图和自由移动 UI。
- 复杂物品栏。
- 长篇开放世界。
- 新增一批 agent 或工具链抽象。

这些以后都可能需要，但它们不是验证「好不好玩」的最短路径。

## 推荐实施顺序

### P0：后端接通 SessionEngine

在 `src/app.py` 增加 session 相关接口，先保留旧 `/invoke`，但不要让前端继续依赖它。

建议接口：

```text
POST /session/start
POST /session/{room_id}/message
POST /session/{room_id}/submit
GET  /session/{room_id}/state
```

建议请求结构：

```json
{
  "room_id": "demo_room",
  "user_id": "user_aria",
  "campaign_id": "whispers_bell_tower",
  "dm_mode": "heuristic",
  "opening": "我推开酒馆的门，走向村长。",
  "random_seed": 20260626
}
```

接口内部做这些事：

- 创建进程级 `SessionEngine` 单例。
- 启动时加载 `canon/*.json` 到 registry。
- `start` 组装一个默认玩家角色和 `campaign_id`。
- `message` 调用 `SessionEngine.message(room_id, user_input)`。
- `submit` 调用 `SessionEngine.submit(room_id, resume_value)`。
- `state` 调用 `SessionEngine.current_state(room_id)`。

返回值直接沿用 `SessionEngine` 的统一 payload，前端先不要二次发明协议。

验收标准：

- 不通过测试脚本，只用 HTTP 能完成 `start → message → submit → combat → finished`。
- `test/test_session_flow.py` 仍能通过。
- 浏览器刷新后能通过 `GET /session/{room_id}/state` 重新拿到当前状态。

### P1：前端做最小跑团界面

替换 Vite starter 页面，先做一个工具型界面。

第一屏布局：

```text
顶部：剧本名、当前地点、状态
左侧/主区域：对话与 DM 叙述时间线
右侧：队伍状态、敌人状态、最近一次检定/战斗结果
底部：玩家输入框
中断浮层：掷骰按钮或行动选项
```

前端状态只需要存：

- `roomId`
- `userId`
- `payload`
- `messages`
- `pendingInterrupt`
- `isLoading`

中断处理：

- `ability_check`、`saving_throw`、`attack_roll`、`initiative_roll`：显示 d20 输入或「掷骰」按钮，提交 `{ "d20": number }`。
- `declare_action`：把后端给的合法 `options` 渲染成按钮，提交选中的行动。
- `damage_roll`：第一版可以自动提交 `{}`，让引擎处理重击补骰。

验收标准：

- 用户不看控制台也能知道下一步该做什么。
- 玩家输入后，页面不会丢失上下文。
- 进入战斗后能看见 HP 变化和当前战斗结果。
- 结局出现时，输入框禁用或显示「重新开始」。

### P2：打磨钟楼短剧本

只围绕 `canon/whispers_bell_tower.json` 打磨，不急着加新剧本。

需要重点验证：

- 开场钩子是否足够清楚。
- 玩家是否知道自己可以去废村。
- 三条线索是否能自然获得。
- 卡关兜底是否会在玩家迷路时出现。
- Boss 战是否有风险但不拖沓。
- 胜利结局是否有反馈感。

如果第一版启发式 DM 太死板，可以先不要全量接 LLM，而是补少量确定性兜底：

- 常见动作到检定类型/DC 的映射表。
- 常见探索动作到 `world_writes.clues_delivered` 的映射。
- 钟楼剧本关键节点的特殊提示。

验收标准：

- 一个新玩家能在 10-15 分钟内玩完。
- 至少经历一次检定和一次战斗。
- 至少看到一个由自己行动导致的世界变化。
- 失败时故事也继续推进，而不是卡死。

### P3：验证 LLM DM 是否增加乐趣

只有在 P0-P2 跑通后，再打开 `dm_mode="llm"` 验证真实模型体验。

验证重点不是「模型聪不聪明」，而是：

- 是否比启发式叙述更有氛围。
- 是否稳定遵守 canon 当前拍。
- 是否会乱改规则或跳剧情。
- 是否会让等待时间明显变长。
- 是否能把失败讲得有意思。

建议保留 `heuristic` 作为基线。每次改 DM prompt 或工具，都用同一条固定流程对比两种模式。

## 好不好玩的判断标准

第一版不要用「功能是否完整」判断成败，要用「玩家是否愿意继续点下一步」判断。

建议观察这些信号：

- 玩家是否能不看说明就知道下一步该输入什么。
- 玩家是否期待掷骰结果，而不是觉得这是表单。
- 战斗中 HP 变化是否带来紧张感。
- DM 叙述是否能承接玩家刚才的行动。
- 检定失败是否仍然产生故事，而不是一句「你失败了」。
- 玩家玩完后是否能复述「我做了什么，所以世界发生了什么」。

如果这些成立，这条路线就是好玩的。否则优先补体验，不要优先补规则覆盖面。

## 近期任务清单

按价值排序：

1. 新增 `SessionEngine` API。
2. 启动时加载 `canon/whispers_bell_tower.json`。
3. 前端替换 starter，做跑团时间线、输入框、中断浮层。
4. 前端显示队伍/敌人 HP。
5. 用浏览器手动跑完一次钟楼剧本。
6. 根据这次手动体验，修 canon、DM prompt 或确定性兜底。
7. 再考虑 LLM DM、多人、更多规则和更多剧本。

## 成功定义

这个阶段完成时，项目应该能做到：

```text
打开网页
  → 点击开始
  → DM 描述灰岩村和钟楼委托
  → 玩家输入自然语言行动
  → 系统在需要时要求掷骰
  → 玩家进入废村探索线索
  → 触发钟楼 Boss 战
  → 玩家选择行动并掷骰
  → 战斗分出胜负
  → DM 给出结局
```

如果这条流程能稳定跑通，下一步再讨论扩展规则和增强 DM 智能才有依据。
