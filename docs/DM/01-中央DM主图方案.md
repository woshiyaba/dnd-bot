# 中央 DM 主图方案（v1·设计稿）

> 状态：**设计稿，未落地**。本文只出方案，等你审核通过后再写实现代码（与 `00-DM节点扩展方案.md` 同一约定）。
>
> 目标：把现在"只在战斗子图里当占位/工具人"的 DM，升级成一个**中央 DM**——她是玩家**唯一**的对话对象，负责跑整个 D&D 世界：
> 探索、社交、调查、旅行、剧情推进；当局势升级为战斗时，她**引导玩家进入战斗子图**，战斗结束后再把世界交回她手里，玩家全程只跟 DM 说话。
>
> 承接 `docs/DM/dm规则.md` 的 DM 职能（演员/导演/即兴/裁判/讲述/构建世界）与 `docs/战斗/README.md` 的铁律：
> **规则归引擎，叙述归 DM，骰子归玩家。** 本方案不破坏这条边界——只是把它从"战斗内"扩展到"整个世界"。

## 二、总体架构（三层图）

用户要的两点——"DM 是一个子图"且"DM 引导进入战斗 graph"——落成**会话主图**编排**两个子图**：

```
会话主图 SessionGraph（顶层，持有 checkpointer，thread_id = session:{room_id}）
  每收到一条玩家消息 = 一次 invoke / Command(resume=…)
  ┌──────────────────────────────────────────────────────────────┐
  │ START                                                          │
  │   → DM 子图 (dm_subgraph)                                       │
  │        ├─ 一般对话/探索/社交 → 直接叙述（DM 可自掷暗骰）→ 回合结束 │
  │        ├─ 需要玩家掷骰 → interrupt 等玩家报骰 → 引擎判定 → 叙述   │
  │        └─ 触发战斗 → 产出 combat scene_context，置 next=combat  │
  │                                                                │
  │   ─(next==combat)─► 战斗子图 (combat_subgraph，复用现有)         │
  │        战斗内部的攻击/先攻/豁免中断照旧（协议同 03 文档）          │
  │        战斗结束 → 把 outcome / 战利品 / 伤亡回灌世界状态           │
  │                                                                │
  │   ─(战斗结束)─► DM 子图（aftermath 模式）叙述战后世界 → 回合结束   │
  │                                                                │
  │   ─(next==wait)─► END（等玩家下一条消息）                        │
  └──────────────────────────────────────────────────────────────┘
```

关键设计：**整个前端只驱动一张图（会话主图）**。无论中断来自 DM 子图（世界检定要玩家掷 d20）还是战斗子图（攻击/豁免骰），都通过**同一条 `interrupt()` → `Command(resume=…)` 通道**冒泡到顶层 `invoke`，前端 resume 协议完全统一（见 §7）。这正是用 LangGraph 子图嵌套的最大好处。

依赖方向（延续既有约束）：`session(主图) → combat → dm → model`，`dm` 不反向依赖 `combat`，`model` 不依赖任何图。

## 三、DM 子图内部设计

DM 子图把"感知 → 决策 → （可选中断）→ 结算 → 叙述"拆成小节点（单一职责，符合 `CLAUDE.md`）：

```
START → perceive → dm_decide ─┬─(intent=reply)──────────────────────────────► END(next=wait)
                              ├─(intent=player_check)→ await_roll(interrupt)
                              │                         → resolve_check(引擎)
                              │                         → narrate_result(DM) ─► END(next=wait)
                              └─(intent=start_combat)─────────────────────────► END(next=combat)
（战后回灌时以 aftermath 入口进入：narrate_aftermath → END(next=wait)）
```

| 节点 | 类型 | 职责 |
|---|---|---|
| `perceive` | 引擎（纯函数） | 组装**最小上下文**：当前场景（地点/在场 NPC/时间/威胁）、最近 N 条对话、本条玩家输入。不下发整个世界状态，控延迟（仿 `dm_bridge._brief`）。 |
| `dm_decide` | DM 智能体 | **大脑**。跑一轮带工具的 DM（骰子 + 知识库），产出结构化"决策"。一般对话时**顺带把叙述流式推给前端**；需要玩家骰子或要开战时只产出意图 + 载荷。 |
| `await_roll` | **中断点** | 仅当 `intent=player_check`。`interrupt(请求)` 等玩家报 d20，请求复用 `build_interrupt_request`（`interrupt_type=ability_check` / `saving_throw`）。 |
| `resolve_check` | 引擎（纯函数） | 用 `validate_d20` 取原始骰 + `ability_check_bonus`/`saving_throw_bonus` 加值，`check_success(d20, bonus, dc)` 判成败。**DM 绝不自己算这个。** |
| `narrate_result` | DM 智能体 | 把检定成败讲成故事（成功→"是,然后…"，失败→"不,但是…"），流式推前端；更新场景。 |
| `narrate_aftermath` | DM 智能体 | 战斗子图返回后专用入口：叙述战后（谁倒下、战利品、世界变化），把 outcome 折回世界状态。 |

> `dm_decide` 与 `narrate_*` 都用现有 `src/dm/agent.py` 的智能体；新增的只是 `dm_decide` 这类"带结构化意图输出"的调用（见 §4），以及把世界上下文喂进去的桥接（仿 `dm_bridge.py`，可新增 `src/dm/world_bridge.py`）。

## 四、核心机制：DM 何时中断、何时自掷（本方案的心脏）

用户的核心诉求："DM 可以控制是否中断；要玩家掷骰就走中断节点等输入；一般情况自己调工具掷骰、和玩家沟通。"
这正是 `dm规则.md` 里 **暗骰 vs 明骰** 与 即兴 **"是,然后…/不,但是…"** 原则的落地。`dm_decide` 输出三选一意图：

| 意图 `intent` | 触发判据（写进系统提示词，让 DM 自己判断） | 走哪条路 |
|---|---|---|
| `reply` | ① 纯叙事/社交/信息，无需检定；② 检定**无意义后果**（`try_again` 规则：可反复尝试且失败无代价 → 直接告诉结果，不掷）；③ 该掷的是**暗骰**（陷阱是否触发、NPC 洞悉对抗、怪物潜行、即兴环境伤害）——DM **用骰子工具自己掷**，把结果编进叙述，玩家无感。 | 直接叙述（DM 可调 `roll_d20`/`roll_expr` 自掷）→ 回合结束 |
| `player_check` | 玩家角色主动做一件**结果不确定且成败都有意义**的事（撬锁、说服、跳跃、豁免陷阱…）。明骰交给玩家，体现公平与参与感。 | `await_roll` 中断 → 引擎判定 → 叙述 |
| `start_combat` | 局势升级到回合制战斗（敌意触发、伏击、玩家宣战）。 | 产出战斗 `scene_context`，置 `next=combat` |

**判据为什么放进提示词而不是写死代码**：要不要掷、明骰还是暗骰，本就是 DM 的"裁判"职能（`dm规则.md`："当规则无法给出明确规定，需要你来做出裁定"）。引擎只兜底**信任边界**——`player_check` 必须真的中断玩家、必须引擎判成败，DM 无权替玩家报骰或直接宣布玩家检定的成败。

**信任边界（不可逾越，延续战斗）**：
- 玩家检定的 d20 **只能**由玩家经 `interrupt()` 报，引擎用 `validate_d20` 钳到 1–20，加值与成败一律引擎算（`resolve_check`）。
- DM 的骰子工具**只服务于**：暗骰、即兴环境伤害、NPC/怪物的随机性、纯叙事检定——**绝不**用于裁定玩家角色的明检定结果，也不碰战斗命中/伤害/HP。
- 这条已在 `src/dm/prompt.py` 的 `DM_BOUNDARY` 写明，本方案在其后补一段"探索期同样适用 + 何时 `player_check`"的判据。

## 五、世界内检定的引擎结算（规则归引擎）

`player_check` 的结算路径与战斗里的豁免/属性检定**同构**，直接复用 `rules.py`：

1. `dm_decide` 产出检定规格：`{ability, dc, proficient?, kind: "ability_check"|"saving_throw", prompt, reason}`。
   - DC 由 DM 依难度裁定（不确定时 `kb_read ability_check` / 即兴伤害严重性表，见知识库）。
2. `await_roll` 用 `build_interrupt_request(kind=InterruptType.ABILITY_CHECK, actor=该玩家, bonus=引擎算好的加值, prompt=…, required_dice="d20")` 发中断。
3. 玩家 resume `{ "d20": 17 }`。
4. `resolve_check`：`bonus = ability_check_bonus(pc, ability, proficient=…)`（或 `saving_throw_bonus`），`success = check_success(d20, bonus, dc)`。
5. `narrate_result` 据 `success` 叙述。

> 这样"世界检定"和"战斗检定"共用同一套规则函数与同一套中断协议，零分叉。新增的只是把检定**主体从'当前回合行动者'换成'对话里的目标玩家'**（探索期没有先攻指针，靠 `directed_to.user_id` 定位）。

## 六、战斗触发与回归（DM ↔ 战斗子图的衔接）

### 6.1 进入：DM 组装 `scene_context`
`dm_decide` 给出 `intent=start_combat` 时，需产出战斗子图能直接吃的 `scene_context`（结构见 `src/model/combat_state.load_combatants` 与 `test/test_combat_flow.build_scene`）：参战者卡面、阵营、`controller`、`surprised`、`loot_table`、`random_seed`。

- 在场怪物/NPC 的卡面从**世界场景的角色册**里取（见 §8 的 `WorldScene.actors`）——DM 不凭空造数值，缺卡时 `kb_read monsters/...`。
- 玩家角色卡来自会话/世界状态里持有的 PC 列表。
- `random_seed` 由主图按房间+遭遇生成，保证可复现。

### 6.2 嵌入方式（两个选项，需你拍板——见 §10）

**选项 A：把战斗编译图当"子图节点"嵌进会话主图（推荐）。**
- 会话主图 `add_node("combat", combat_compiled_graph)`，战斗内部的 `interrupt()` **自动冒泡**到顶层 `invoke`，前端 resume 通道天然统一。
- 代价：① 战斗子图**不能再自带 checkpointer**——改由会话主图统一持有（见 §9）；② 两图状态 schema 不同（`DMState` vs `CombatState`），需一个**包装节点**做 `DMState→输入CombatState` 与 `CombatState→回灌DMState` 的映射。
- 需改 `build_combat_graph`：支持"**不挂 checkpointer**、仅返回 builder/可嵌入图"的模式（当前它缺省塞了 `MemorySaver`）。

**选项 B：会话主图的 `combat` 节点内部调 `CombatEngine`（façade）。**
- 复用现成 `CombatEngine`，战斗用**独立 thread_id**（`combat:{room}`）自管 checkpointer。
- 代价：战斗的中断**不会**冒泡成会话主图的中断——会话节点得在内部自驱"战斗中断↔前端"的来回，等于在主图里再实现一遍 03 文档的循环，前端要区分"现在 resume 的是哪张图"。组合性差。
- 仅当 A 的子图嵌套在当前 LangGraph 版本上踩坑时，作为回退。

> 倾向 **A**：它才真正实现"前端只跟一张图说话""DM 引导进入战斗 graph"的无缝体验。B 留作回退。

### 6.3 回归：战斗结果回灌世界
战斗子图 `settle` 后返回 `outcome` / 写回（HP、`granted_loot`、`writeback`）。会话主图把这些折回 `DMState`（更新 PC 的 HP/物品、场景的伤亡与战利品），再以 **aftermath 入口**进 DM 子图，由 `narrate_aftermath` 讲战后。之后 `next=wait`，玩家继续只跟 DM 对话。

## 七、中断协议复用（前端零改协议）

所有中断（世界检定 + 战斗骰）共用 `docs/战斗/03` 的请求/恢复值格式与 `directed_to.user_id` 路由。新增/澄清点：

| 项 | 说明 |
|---|---|
| 复用中断类型 | 探索期玩家检定用现有 `ability_check` / `saving_throw`，**无需新增枚举**。 |
| `directed_to` | 探索期没有先攻指针，`actor` = 对话所指向的玩家角色；其 `controller` → `user_id`，前端据此推给本人。 |
| 旁观广播 | 单人先跑通；多人时其余玩家收"等待 X 进行 ××检定"的旁观消息（同 03 §5）。 |
| 叙述推流 | `dm_decide` / `narrate_*` 复用 `StreamCollector` + `get_stream_writer()` 的 custom 通道（与 `agent.dm_narrate`、`graph.py` 一致），前端按 `{node,status,chunk}` 解析，**协议不变**。 |
| 顶层驱动 | `app.py` 新增会话入口（如 `POST /session/{room}/message` 起对话、`POST /session/{room}/resume` 报骰），内部对**同一** `session:{room}` thread 跑 `ainvoke` / `Command(resume=…)`，与 `CombatEngine._interpret` 同款"归一负载"。 |

## 八、状态对象设计（新增 `DMState` / `WorldScene`）

会话主图需要一个比 `CombatState` 更上层的持久状态（按 `session:{room_id}` 存档）：

```python
class DMState(TypedDict, total=False):
    messages: list            # 玩家↔DM 对话历史（截断/摘要喂给 perceive）
    scene: dict               # 当前世界场景 WorldScene（见下）
    party: dict               # 玩家角色册：pc_id -> 角色卡（HP/物品跨场景延续）
    pending_check: dict | None  # dm_decide 决定的待掷检定规格（ability/dc/kind/actor…）
    last_check_result: dict | None
    combat_request: dict | None # 触发战斗时 DM 组装的 scene_context
    last_combat_outcome: dict | None
    next: str                 # 路由信号：wait | combat | aftermath
    room_id: str
    campaign_log: list        # 全程世界事件流（前端回放 + DM 长期上下文）
```

`WorldScene`（`scene` 的形状，给 DM 的"环境感知"）：

```python
{
  "location": "废弃神殿·前厅",
  "description": "……（DM 维护的环境描述）",
  "actors": [ { "actor_id", "name", "disposition": "hostile|neutral|friendly",
                "card": {…可转战斗卡面…} } ],   # 在场 NPC/怪物
  "exits": ["东门", "地下室"],
  "flags": {"火盆已点燃": true},
  "threat": "潜在伏击" | None
}
```

> `model/` 仍不认识 LangGraph。`DMState`/`WorldScene` 放 `src/model/`（数据形状）或 `src/dm/`（若只主图用）——倾向 `src/model/dm_state.py`，与 `combat_state.py` 并列，保持"数据形状归 model"。

## 九、Checkpointer 与 thread 统一（选项 A 的技术关键）

当前 `build_combat_graph` 缺省自带 `MemorySaver(serde=_build_serde())`，`_COMBAT_SERDE_WHITELIST` 登记了战斗模型类。改造为子图嵌套后：

1. **会话主图持有唯一 checkpointer**（`thread_id = session:{room}`）；战斗子图**不再自挂** checkpointer（编译为可嵌入图）。
2. `_build_serde()` 的**白名单上移**到会话主图的 checkpointer——`DMState` 里也会持久化战斗模型对象（party 卡面），白名单需覆盖 `combat_state` 那一组类（直接复用 `_COMBAT_SERDE_WHITELIST` 即可，必要时并入 PC/世界相关类）。
3. 多人/重启：把 `MemorySaver` 换成持久化版（SQLite/MySQL saver），与 03 文档 §5 要求一致。

> 落地细节按当时 LangGraph 版本对"子图共享父 checkpointer / 跨 schema 子图"的支持验证；若版本受限则回退 §6.2 选项 B。
