# DM 节点扩展方案（v0·待审核）

> 目标：给战斗子图里的 **DM 节点**装上两样东西——
> 1. **骰子工具**：`d4/d6/d8/d10/d12/d20`，让 DM（LLM）在需要时自己掷骰判定；
> 2. **轻量「扩展查阅」机制**：一个本地 Markdown 知识库（规则 / 怪物 / 技能），DM 按需检索、按需读取，**不把整本规则塞进每轮提示词**，以此压低延迟。
>
> 硬约束（来自你的要求）：**不使用 deepagents**（太重），只用 **LangChain 原生 API**（`ChatOpenAI.bind_tools` + 自写工具循环）。
> 本文只出**方案**，等你审核通过后再写实现代码。

承接 `docs/战斗/README.md` 的核心原则：**规则归引擎，叙述归 DM，骰子归玩家**。本方案**不改动**这条边界——DM 的骰子只用于**叙事/判定/怪物决策**，命中、伤害、HP 仍由引擎在 `rules.py` 里算。

---

## 一、DM 现在在哪（落点）

DM 目前是三个**确定性占位**节点（见 `src/combat/nodes.py`），本方案把它们升级为「可选接 LLM」：

| 节点 / 位置 | 现状（占位） | 升级后（LLM 模式开启时） |
|---|---|---|
| `judge_surprise` | 读 `scene_context["surprised"]` 名单 | DM 用「潜行 vs 被动察觉」对抗判定，可掷骰、可查规则 |
| `declare_action` → `_dm_decide` | 「打血最低的够得着的敌人」启发式 | DM 读怪物卡/技能库，决定怪物动作与目标 |
| `narrate` | 模板拼句 | DM 把本回合事件讲成有氛围的故事（流式推送） |
| `_resolve_improvise` 的 DC | 默认 12 | DM 依「即兴伤害/严重性」表给 DC（查规则库） |

> **不动的部分**：`resolve_action` 的命中/伤害/HP 结算、玩家中断收集、先攻排序——全部仍是纯引擎确定性代码。

## 二、整体结构（新增 `src/dm/` 包）

DM 是比战斗更上层的概念（`README` 提到战斗子图日后会成为「DM 主图」的子图），所以独立成包，供 `src/combat` 调用，方向是 `combat → dm`，不反向依赖：

```
src/dm/
  __init__.py
  knowledge.py   # 知识库注册表：扫描 md、解析 frontmatter、建目录、读正文
  tools.py       # LangChain @tool：骰子工具 + 知识库检索/读取工具
  prompt.py      # DM 人设与「通用规则」基线提示词（常驻，不查阅）
  agent.py       # 轻量工具循环：bind_tools + 手写 tool-call 循环，缓存模型
```

知识库内容目录（运行期内容，独立于 `docs/` 设计文档）：

```
knowledge/
  rules/      # 检定的「特定规则」条目（从 docs/DM/dm规则.md 拆出）
  monsters/   # 怪物卡 md（你后续补充）
  skills/     # 技能 md（你后续补充）
```

> 为什么内容放 `knowledge/` 而不是 `docs/`：`docs/` 是给人看的设计稿；`knowledge/` 是给 DM 运行时查的数据，二者职责不同。`docs/DM/dm规则.md` 仍保留为「规则总稿」，KB 条目从它派生。

## 三、知识库与「扩展查阅」机制（核心）

### 3.1 文档格式：带 frontmatter 的 Markdown（仿 SKILL.md，但自己解析）

```markdown
---
id: rule_group_check
name: 团体检定
category: rule          # rule | monster | skill
tags: [检定, 队伍]
description: 队伍共同达成某事、强者掩护弱者时的判定法；半数以上成功即通过。
---

（正文：完整规则文字，DM 只在需要时才读取这一段）
```

- `description` 是**给 DM 看的一句话摘要**，进目录；`正文`只在 `kb_read` 时才返回。
- 解析用极轻量的 frontmatter 切分（`---` 分隔 + `yaml.safe_load`，或手写 KV 解析，避免引重依赖）。

### 3.2 注册表 `knowledge.py`

启动时扫描 `knowledge/**.md`，建立索引 `{id -> (path, frontmatter)}`；正文**惰性读取**（首次 `kb_read` 才读盘并缓存）。提供：

- `catalog(category=None) -> list[{id, name, category, description}]`：轻量目录，可全量放进系统提示词（每条只一句话）。
- `search(query, category=None) -> list[catalog项]`：关键词 / tag 命中。
- `read(id) -> str`：返回某文档正文（带缓存）。

### 3.3 暴露给 DM 的工具（`tools.py`）

| 工具 | 签名 | 用途 |
|---|---|---|
| `kb_search` | `(query: str, category: str = "")` → 目录 JSON | DM 发现「有没有相关规则/怪物/技能」 |
| `kb_read` | `(doc_id: str)` → 正文 | DM 确认要用某条后，读全文 |

**延迟策略（回应你的诉求）**：
- **通用规则**（DM 职能、即兴原则「是,然后…」、节奏把控等）→ 写死进 `prompt.py` 系统提示词，**常驻、零查询**。
- **特定规则**（团体检定、被动检定、即兴伤害表、再次尝试…）→ 进 KB，**目录常驻（每条一句话）、正文按需读**。
- 这样每轮请求基线很小，只有 DM 判断「这局面要用某条特定规则」时才多一次 `kb_read`，而不是每轮把整本规则一起送进去。

### 3.4 从 `docs/DM/dm规则.md` 拆出的初始 KB 条目（rules/）

| id | name | 来源段落 |
|---|---|---|
| `rule_improvise_damage` | 即兴伤害表 | 「即兴伤害 Improvising Damage」 |
| `rule_damage_severity` | 伤害严重性与等级 | 同上的等级对照表 |
| `rule_ability_check` | 属性检定 | 「属性检定」 |
| `rule_proficiency` | 熟练加值适用 | 「熟练 Proficiency」 |
| `rule_try_again` | 再次尝试 | 「再次尝试」 |
| `rule_group_check` | 团体检定 | 「团体检定」 |
| `rule_passive_check` | 被动检定 | 「被动检定」 |
| `rule_attack_roll` | 攻击检定 | 「攻击检定」 |
| `rule_saving_throw` | 豁免检定 | 「豁免检定」 |

> 怪物/技能条目由你后续补充，注册表零改动即可识别（约定好 frontmatter 字段即可）。

## 四、骰子工具（`tools.py`）

给 DM 绑定 6 个掷骰工具，**复用引擎的可复现 `Dice`**（不另起随机源）：

| 工具 | 说明 |
|---|---|
| `roll_d4 / d6 / d8 / d10 / d12 / d20` | 各掷一颗，返回原始点数 |
| （可选）`roll_expr(expression)` | 掷 `NdM+K` 表达式，复用 `dice.parse`/`Dice.roll` |

实现要点：
- 用**工厂** `build_dice_tools(dice: Dice)` 闭包持有引擎当前的 `Dice` 实例，保证 DM 掷骰与引擎掷骰**同一可复现序列**（同 `random_seed` 可回放）。
- 工具返回结构化结果（点数 + 表达式），便于 DM 在叙述里引用，也便于日志。

### 边界守则（重要）
- DM 的骰子**只服务于**：突袭对抗、即兴动作 DC 判定、怪物决策的随机性、纯叙事检定。
- DM **绝不**用这些骰子去算「命中/伤害/扣血」——那条路径仍走 `resolve_action` + `rules.py`，玩家的攻击骰仍走 `interrupt()`。这条在系统提示词里也会明确写给 DM。

## 五、DM 运行器（`agent.py`，纯 LangChain，无 deepagents）

手写「工具调用循环」，不引 agent 框架：

```python
model = create_chat_model(model=DM_MODEL)        # 复用 llm_util，DashScope 兼容
llm = model.bind_tools(dice_tools + kb_tools)
messages = [SystemMessage(base_prompt), HumanMessage(task_context)]
while True:
    ai = await llm.ainvoke(messages)
    messages.append(ai)
    if not ai.tool_calls:
        break
    for call in ai.tool_calls:                    # 串行执行工具，回填 ToolMessage
        out = tools_by_name[call["name"]].invoke(call["args"])
        messages.append(ToolMessage(out, tool_call_id=call["id"]))
return ai.content                                 # 决策走 JSON，叙述走纯文本
```

- **决策类**（突袭、怪物动作、DC）→ 要求模型最终输出 JSON，用现有 `src/common/utils/json_parser.extract_json_object` 防御式解析成 `current_action` / 判定结果。
- **叙述类**（`narrate`）→ 最终文本通过 `get_stream_writer()` 推 `{node:"narrate", status, chunk}`，**复用现有 custom 事件通道**（与 `graph.py`、占位 `narrate` 一致），前端无需改协议。
- **缓存**：仿 `example_agent` 的 create-once 模式——按系统提示词缓存 `bind_tools` 后的模型；知识库目录变化时失效重建。

> `with_structured_output` 在「同时要工具 + 结构化输出」时和 DashScope 兼容端点配合不稳，故选**手写循环 + JSON 解析**，更可控。

## 六、接入战斗子图（开关 + 回落）

为保持「战斗可脱离 DashScope 独立跑」，LLM 版 DM 做成**可开关**，缺省回落现有占位：

- `scene_context["dm_mode"]`：`"heuristic"`（默认，现状占位）| `"llm"`（启用 DM 运行器）。
- 也可在 `"llm"` 但无 `DASHSCOPE_API_KEY` 时自动回落 `"heuristic"`，并记一条 warning。
- 涉及 LLM 的节点（`judge_surprise` / `declare_action` / `narrate`）改为 `async def`：LLM 分支走 `await dm_agent(...)`，启发式分支保持同步逻辑。`interrupt()`（玩家分支）与 async 节点共存无碍。

每个 DM 节点调用运行器时传入**最小上下文**（当前行动者、存活敌我、本回合事件、可选轮次/区域），不下发整个 `CombatState`，进一步控延迟。
