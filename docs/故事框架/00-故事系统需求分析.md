# 故事系统需求分析（v0·先理需求，不写代码）

> 状态：**需求/方案稿**，未落地。按项目惯例先出方案、审核通过再写实现。
> 立意（来自你的构想）：**故事是核心，战斗与探索都为故事服务；DM 本质是给玩家讲故事。**
> 一句话原则：**骨架要提前定死，血肉要现场长出来。**
> 形状：**糖葫芦**——主线是一串基本线性的「珠子」（保证有头有尾、一定能结束），
> 每颗珠子内部是一个可自由探索的小沙盒（玩家怎么晃都行，只有触动这颗珠子的「推进条件」才串到下一颗）。
> 参考流程图：`docs/故事框架/image.png`（开场·任务引入 → 探索·冲突〔可往复〕→ 高潮·Boss决战 → 结局）。

---

## 一、为什么需要它（现状的洞）

当前 `src/session` + `src/dm` 已能跑「一个对话回合」，但**没有故事**：

- `DMState` 只持有 `scene`（单个当前场景）+ `messages`（最近若干对话），DM 的全部认知就这么多。
- `world_bridge.decide_turn` 是**纯反应式**：读「场景+对话+这步输入」决定 `reply/player_check/start_combat`，
  **没有目标、没有进度、没有出口、没有结束条件**。
- 结果：玩家能在一个 `scene` 里无限打转，**故事不会开始也不会结束**——这就是「缺少灵魂、缺少剧本」。

本系统补的正是这条主轴：给 DM 一份**预先定死的剧本骨架（canon）**，让每一回合的自由叙述都长在骨架上，
并把「触发就推进」从感觉变成**引擎可判定的逻辑**。

> 同时回应 `docs/待改进问题.md`：
> - **问题 1（裁判方差大）**：推进不再靠 DM 自由发挥，而是引擎按预写的「推进条件」判定；语义判定被收窄成「某条固定条件是否满足」的是/否题，方差大幅下降。
> - **问题 2（核心乐趣未验证）**：有了剧本骨架，才谈得上「一场有头有尾的冒险好不好玩」，验证才有对象。

## 二、核心概念：两层「定死」与「现场长」

| | 提前定死（canon·剧本圣经） | 现场长出（DM 临场生成） |
|---|---|---|
| 是什么 | 大纲、每一拍的关键节点、主要地点、重要 NPC 及其**目标/秘密**、Boss、每拍的**推进条件**与**必须传达的关键线索**、整局**胜负条件** | 真正的叙述文字、NPC 临场对话、对玩家奇葩操作的反应、边角描写 |
| 谁负责 | **故事生成系统**（离线，一次性生成并冻结）或人工编写 | **DM 智能体**（每回合，拿骨架+当前状态即兴） |
| 保证 | 故事**一定有头有尾、一定能结束** | 代入感、临场感、灵活接住玩家创意 |
| 在引擎里 | 只读数据（`Canon`），随局冻结 | 走现有 `reply / player_check / start_combat` 机制 |

**铁律延伸**：现有「规则归引擎，叙述归 DM，骰子归玩家」**再加一条——「结构归引擎」**：
**拍与拍之间的推进只能由引擎依据 canon 的推进条件判定，DM 无权跳拍或改写骨架。**
DM 只能在当前这颗珠子的沙盒里自由叙述。

## 三、数据结构需求（提前定死的那部分）

> 数据形状归 `model` 层（graph-free），建议新增 `src/model/canon.py`。英文标识符 + 中文注释，沿用项目规范。

### 3.1 `Canon`（剧情圣经·整局冻结）
- `title` 标题、`premise` 一句话主线、`theme/tone` 主题基调
- `win_condition` / `lose_condition`：整局**胜利/失败条件**（达成即进「结局」拍）
- `cast: list[NpcSpec]`：重要 NPC / Boss 册
  - `NpcSpec`：`id, name, role, goal(目标), secret(秘密), disposition, card?(可转战斗卡面)`
- `locations: list[LocationSpec]`：主要地点
  - `LocationSpec`：`id, name, description, intra_exits(珠内地点互通)`
- `beats: list[Beat]`：主线珠子串（见 3.2）
- `start_beat_id`：起始拍

### 3.2 `Beat`（一拍 / 一颗糖葫芦珠）——这是「触发推进」可判定的关键
每一拍**至少**包含：
- `id` / `title` / `kind`（`opening | exploration | conflict | climax | ending`，对应流程图四段）
- `location_ids: list[str]`：这颗珠子内的沙盒地点（**可多个**，珠内自由探索）
- `entry_state`：进入这拍时世界初始状态（在场 NPC、布景、初始 flags）
- `key_info: list[KeyInfo]`：**DM 必须让玩家获知的关键线索**（每条带「是否已传达」标记）
- `advance_conditions: list[Trigger]`：**推进条件**（玩家做到什么 / 什么为真，才离开这颗珠子）
- `exits: list[Exit]`：出口 `trigger_id → next_beat_id`（**通向哪一拍或哪几拍**）
- `stuck_fallback`：**卡关兜底**——迷路/空转太久时给什么提示（指向哪个出口、暴露哪条线索）
- `encounter?`：若是 conflict/climax 拍，预置遭遇模板（参战者、阵营、`loot_table`、`random_seed`）

### 3.3 `Trigger`（推进条件）——把「触发」变成可判定
每个触发器声明**判定方式**，引擎据此决定能否自动判：

| `kind` | 判定方 | predicate 示例 |
|---|---|---|
| `flag` | **引擎**（确定） | `{flag: "拿到密钥", equals: true}` |
| `item` | **引擎**（确定） | `{item_id: "古钥匙", holder: "party"}` |
| `location` | **引擎**（确定） | `{location_id: "地下祭坛"}`（玩家到达） |
| `combat_outcome` | **引擎**（确定） | `{encounter_id: "boss", outcome: "players_win"}` |
| `semantic` | **DM 是/否题**（兜底） | `{prompt: "玩家是否说服守卫放行？"}` |

> 设计要点：**能编码成 flag/item/location/combat 的，绝不留给语义判定**。语义触发只用于「确实无法编码的剧情判断」，
> 且被收窄成「针对一条**预写好的固定条件**回答是/否」——比让 DM 开放式裁定可靠得多（直接缓解问题 1）。

### 3.4 故事进度（`DMState` 新增字段，随 checkpoint 持久化）
- `canon`：本局冻结的剧情圣经（或其引用 id，见决策 #2）
- `story`：进度工作区
  - `current_beat_id`：当前在哪颗珠子
  - `visited_beats: list[str]`：已走过的拍
  - `flags: dict[str, Any]`：世界 flag（推进条件的判定依据）
  - `delivered_clues: list[str]`：已传达的关键线索 id
  - `beat_entered_turn` / `idle_turns`：进入本拍的回合数 / 在本拍空转回合数（驱动卡关兜底）

## 四、运行机制需求（现场长 + 触发推进）

### 4.1 「触发推进 / 否则探索」——一个确定性节点
**每个 DM 回合结束后**，引擎做一件确定的事：检查玩家这步**有没有满足当前拍的某个推进条件**。

```
玩家这一步（reply / 检定结算完 / 战斗结束）
        │
        ▼
  evaluate_advancement（引擎为主）
    1) 先用 flag/item/location/combat_outcome 触发器，对世界 flag 做确定性判定
    2) 仅剩的 semantic 触发器 → 问 DM 一道是/否题（窄判定）
        │
   ┌────┴─────────────┐
   触发命中            未命中
   │                   │
   ▼                   ▼
 enter_beat         留在原地
 （切到出口指向的下一拍）   （继续珠内自由探索叙述）
```

- 命中 → `enter_beat`：加载下一拍 → 用 `entry_state` 搭好新场景（`scene`）→ DM 叙述「进入新珠子」的过场 → 重置 `idle_turns`。
- 未命中 → 停在本拍，`idle_turns += 1`，DM 继续按当前拍目标自由叙述。
- 这正是流程图里「探索·冲突〔可往复〕」那条**自循环**：未触发就一直在珠内打转。

### 4.2 让 DM 的叙述「长在骨架上」（增强现有 DM 节点）
现有 `perceive` 只喂「场景+对话+输入」。**增强为同时喂当前拍的骨架**：
- 本拍目标（要把玩家引向哪个出口）、尚未传达的 `key_info`、本拍在场 NPC 的目标/秘密、可用出口的提示。
- 这样 DM 即兴时**心里有方向**：自由接话，但自然朝推进条件靠拢——「不硬拽玩家，但故事在往前走」。

### 4.3 卡关兜底（`stuck_fallback`）
- `idle_turns` 超过阈值 → 给 DM 追加一条指令：**主动抛出一条尚未传达的 `key_info`，或按 `stuck_fallback` 给指向出口的提示**。
- 保证玩家不会因为「不知道下一步」而卡死整局，同时不破坏沙盒的自由感。

### 4.4 结局与胜负
- 每回合（尤其战斗后）检查 `win_condition` / `lose_condition`：
  - 胜利 → 进「结局·胜」拍，DM 收尾，按 flag 决定具体结局文案（流程图「按 flag 收尾，可开新局」）。
  - 失败（如团灭 TPK）→ 进「结局·败」拍。
- 到达 `kind=ending` 的拍并叙述完 → 整局 `END`（可开新局）。

### 4.5 flag 谁能写（守住「结构归引擎」）
- **引擎自动写**：战斗结算（`boss_dead`）、玩家移动到某地点、拾取某道具。
- **DM 受限声明**：DM 在 `reply/检定` 时可附带 `flags_set`，但**只能写 canon 预先声明存在的 flag**（白名单校验）。
  防止 DM 凭空造状态导致剧情漂移。

## 五、故事生成系统需求（外围·离线一次性）

> 这是「提前定死」的生产者，**不在每回合的热路径上**。建议新增 `src/story/`（依赖方向：`session → story`，`story → model`，不反向）。

需要的能力：
1. **`generate_canon(seed)`**：输入种子（主题/基调、队伍等级人数、目标长度〔几颗珠子〕）→ 跑一轮 LLM「编剧」→ 产出 `Canon` JSON。
   - 复用现有 `create_agent` / `dm_complete_json` + `extract_json_object` 的稳健解析，不引新重依赖。
2. **`validate_canon(canon)`**（纯函数，放 `model/canon.py`）——**这是「一定能结束」的结构性保证**：
   - 每颗珠子从 `start_beat_id` **可达**；非 ending 拍**至少有一个出口**；ending 拍**可达**。
   - 所有 `exits.next_beat_id` / 引用的 `location_id` / `npc_id` / `flag` 都**真实存在**（无悬空引用）。
   - `win_condition` / `lose_condition` 已定义。
   - **校验不过就不放行**——把「有头有尾」从「祈祷」变成「编译期断言」。
3. **人工编写同等可行**：`Canon` 只是数据，生成器是便利工具；先**手写一个 canon 跑通全链路**，生成器作为二期。

## 八、建议的落地顺序（审核通过后）

1. **`src/model/canon.py`**：定 `Canon/Beat/Trigger` 形状 + `validate_canon` + 确定性触发判定。
2. **手写一个最小 canon**（3~4 拍：开场→探索→Boss→结局），跑 `validate_canon` 确认结构闭合。
3. **`DMState` 加 `canon/story`**；`session` 插入 `evaluate_advancement / enter_beat` 节点（先只支持 flag/location/combat 触发，语义二期）。
4. **增强 `perceive`**：把当前拍骨架喂给 DM，让叙述长在骨架上。
5. **接 `judge_trigger` 语义判定 + `stuck_fallback` 卡关兜底**。
6. **冒烟脚本** `test/test_story_flow.py`：真玩家驱动「开场→珠内探索→触发推进→Boss战→结局」全链路，验证「一定能结束」。支持自动驱动，玩家驱动的话命令行交互
7. **`src/story/generator.py`**：LLM 编剧（二期）。

> 验证目标始终是那一句：**拿着这副骨架，让真模型把一整局冒险讲完，读下来有头有尾、且好玩。**
