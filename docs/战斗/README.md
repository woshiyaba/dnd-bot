## 一、为什么用 LangGraph（框架选型结论）

**结论：用 LangGraph，不换框架。** 理由：

1. **`interrupt()` 原生支持人在回路**——正是"骰子交给玩家"和"玩家声明动作"的需求。图执行到中断点会挂起并返回中断负载；前端把它推给对应玩家，玩家报数后用 `Command(resume=值)` 恢复，**战斗状态全程不丢**。
2. **`StateGraph` + 条件边** 天然表达回合循环：`检查结束 →(否) 下一位 / →(是) 结算`，正好是 `image.png` 里那条"否，回到上一位的下一位"的回边。
3. **Checkpointer 持久化整场战斗**——按"房间"用唯一 `thread_id` 存档，多人、跨多条消息、服务重启都不会把这场仗弄丢。**这是多人战斗的刚需。**
4. **与现有栈一致**（项目已在用 LangGraph + deepagents + FastAPI + WebSocket）。战斗子图可作为日后 DM 主图的一个**子图**被调用，符合"先做战斗、再做 DM"的规划。
5. **引擎/DM 分工能干净落地**：引擎节点 = 纯 Python 确定性代码（算命中、扣 HP、排先攻）；DM 节点 = 调 LLM（判突袭、替怪物决策、叙述结果）。

### 落地前要改的两处现状（来自现有 `graph.py`）

- 现在用 `MemorySaver` + 硬编码 `thread_123`。多人战斗必须换**持久化 checkpointer**（SQLite / MySQL / Redis 皆可），并用**每场战斗唯一的 `thread_id`**（建议 `combat:{房间id}`）。
- 单个 `thread` 同一时刻只处理一个中断——这反而是好事：按先攻顺序**串行**收集玩家输入，不需要处理并发输入。

## 二、核心设计原则（贯穿所有文档）

**规则归引擎，叙述归 DM，骰子归玩家。**

| 谁 | 负责 | 在图里体现为 |
|---|---|---|
| **引擎**（确定性代码） | 排先攻、命中判定、扣 HP、加状态、判胜负、写回世界 | 普通 Python 节点 |
| **DM**（大模型） | 判突袭、替怪物选目标和动作、叙述本回合 | 调 LLM 的节点 |
| **玩家** | 掷自己的骰子（先攻 / 攻击 / 伤害 / 豁免）、声明动作 | `interrupt()` 中断点 |

> 绝不让 LLM 算命中或记 HP——那是引擎的活，必须是状态里的确定数字。
> 玩家的骰子靠中断收集；**怪物与环境的骰子由引擎自动掷**（用可复现的随机源）。

## 三、流程一览（详见 02 文档）

```
START
  → 进入战斗(引擎)          # 初始化 CombatState、摆好区域
  → 判定突袭(DM)            # 标记被突袭方，其首回合被跳过
  → 掷先攻(引擎+玩家中断)    # 玩家报 d20，怪物引擎掷 → 排出行动顺序
  ┌─────────────────────────────────────────────┐
  │ → 下一位(引擎)           # 推进先攻指针，处理跳过/回合开始结算  │
  │ → 声明行动(玩家中断/DM)  # 玩家选攻击·技能·道具·创意；怪物DM决定 │
  │ → 结算行动(引擎+玩家中断) # 玩家报攻击/伤害骰，引擎比对AC、扣HP   │
  │ → 叙述(DM)              # 把本回合事件讲成故事                │
  │ → 检查结束?(引擎条件边)                                      │
  │      ├─ 否 ───────────────────────────────────► 回到"下一位" │
  │      └─ 是 ─► 结算(引擎)  # 写回 HP/战利品/flag → END         │
  └─────────────────────────────────────────────┘
```

绿色 = 引擎自动；紫色 = DM（大模型）；带"中断"= 需要玩家输入。
