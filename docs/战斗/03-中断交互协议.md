# 03 · 中断交互协议（骰子交给玩家）

> 玩家要掷的骰子，全部用 LangGraph `interrupt()` 收集。本文定义：**在哪些点中断、请求长什么样、玩家怎么回、怎么和现有 WebSocket 对接**。
> 节点位置见 [`02-LangGraph流程设计.md`](./02-LangGraph流程设计.md)。

---

## 1. 中断点清单

| 中断类型 | 触发节点 | 向谁要 | 要什么 |
|---|---|---|---|
| `掷先攻` | `roll_initiative` | 每个玩家 | 一个 d20 原始值 |
| `声明行动` | `declare_action` | 当前回合玩家 | 选哪个动作 + 目标 |
| `攻击检定` | `resolve_action` | 攻击方玩家 | 攻击 d20 原始值 |
| `伤害掷骰` | `resolve_action` | 攻击方玩家 | 伤害骰结果（命中后；重击则翻倍骰数） |
| `豁免检定` | `resolve_action` | 被作用方玩家 | 豁免 d20 原始值 |
| `属性检定` | `resolve_action` | 行动方玩家 | 创意动作的检定 d20 |

> 怪物/环境的同类骰子**不中断**，由引擎自动掷。

---

## 2. 中断请求格式（图 → 前端）

节点里 `value = interrupt(请求)`，`请求` 统一为：

```python
{
    "中断类型": str,        # 见上表
    "面向": {
        "combatant_id": str,
        "user_id": str,     # = Combatant.操控者，前端据此推给正确的人
    },
    "提示": str,            # 给玩家看的话，如 "轮到奥德里克，掷先攻：d20 + 1"
    "需要骰子": str | None, # 骰子表达式，如 "d20" / "1d8+3"；纯选择类为 null
    "加值": int,            # 引擎会替你加的固定值（前端可显示 "+1"）
    "选项": list | None,    # 仅"声明行动"用：可选动作与合法目标
    "期望返回": dict,       # 该类型恢复值的 schema（见第 3 节）
}
```

### 2.1 `声明行动` 的 `选项` 结构

```python
"选项": {
    "攻击": [
        {"攻击名":"长剑","目标":[{"id":"mob_goblin_1","名字":"哥布林","区域":"前排"}]},
        {"攻击名":"手弩","目标":[...]},   # 已按射程/区域过滤掉够不着的
    ],
    "技能": [{"技能id":"skill_second_wind","名字":"重振旗鼓","剩余充能":1}],
    "道具": [{"道具id":"item_healing_potion","名字":"治疗药水","数量":2}],
    "移动": [{"目标区域":"后排"},{"目标区域":"门口"}],   # 可选
    "创意": true,   # 允许自由文本
}
```

## 3. 恢复值格式（前端 → 图）

前端收集玩家输入后，用 `graph.invoke(Command(resume=恢复值), config={thread_id})` 恢复。
各中断类型的 `恢复值`：

```python
# 掷先攻 / 攻击检定 / 豁免检定 / 属性检定
{ "d20": 17 }                    # 玩家报的 d20 原始值；引擎自己加"加值"

# 伤害掷骰
{ "结果": 7 }                    # 玩家把整条伤害骰算好的总和报上来

# 声明行动（攻击示例）
{ "行动类型":"攻击", "攻击名":"长剑", "目标id":"mob_goblin_1" }
# 声明行动（技能/道具/创意/移动）
{ "行动类型":"技能", "技能id":"skill_second_wind", "目标id":"self" }
{ "行动类型":"道具", "道具id":"item_healing_potion", "目标id":"self" }
{ "行动类型":"创意", "描述":"我踹翻火盆把火星溅向哥布林" }
{ "行动类型":"移动", "目标区域":"后排" }   # 移动后通常还需再发一次"声明行动"收主要动作
```

> **信任边界**：玩家只报 `d20` 原始值，**加值与命中判定一律引擎算**——前端别替玩家算总和，避免作弊和口径不一致。可做范围校验（1–20）。

## 4. 减少往返：攻击"一次中断收齐"（推荐）

一次玩家攻击天然有 2 次掷骰（命中、伤害）。逐个中断 = 2 个往返，多人会很慢。优化：

- `攻击检定` 的请求里把伤害骰也带上，让玩家**同时报攻击和伤害**：
  ```python
  请求 = { "中断类型":"攻击检定", ..., "需要骰子":"d20", "附带":{"伤害骰":"1d8+3"} }
  恢复值 = { "d20":17, "伤害结果":7 }   # 未命中则引擎忽略 伤害结果
  ```
- 引擎用 `d20` 判命中，命中才采用 `伤害结果`，未命中丢弃。
- 重击（d20=20）需要翻倍骰数时，引擎检测到后**补一个** `伤害掷骰` 中断要额外的骰子（重击较少，单独要可接受）。

> 这是 UX 取舍，不影响规则正确性。先做"逐个中断"最简单，跑通后再合并。

## 5. 与现有 WebSocket / FastAPI 对接

现项目（见 `app.py` / `graph.py` / `ws_manager.py`）已有：`/ws/{user_id}` 每用户连接、`manager` 按 `user_id` 推送、`astream_agent_collect` 推流 token。战斗中断接进去：

```
玩家A 连 /ws/A、玩家B 连 /ws/B …（同一房间）
            │
POST /combat/{房间id}/action  → graph.invoke(Command(resume=恢复值),
            │                                 config={"thread_id": f"combat:{房间id}"})
            ▼
图推进到下一个 interrupt() → invoke 返回带 __interrupt__ 的结果
            │
读 中断请求.面向.user_id → manager 把"该谁掷什么"推给那个玩家的 ws
            │
该玩家前端弹"掷 d20"→ 回报 → 再次 POST /combat/{房间id}/action 恢复
```

要点：
1. **`thread_id` 按房间唯一**（`combat:{房间id}`），替换现有硬编码 `thread_123`；checkpointer 换持久化版（SqliteSaver/自建 MySQL saver），否则重启丢档。
2. **路由**：从返回的中断负载取 `面向.user_id`，只推给该玩家；其余玩家收一条"等待 X 行动"的旁观消息即可。
3. **叙述推流**：`narrate` 节点继续用 `astream_agent_collect` 把 DM 叙述 token 流给**全房间**。
4. **重连**：因为状态在 checkpointer，玩家断线重连后，用同一 `thread_id` 重新取当前中断负载即可恢复"该你掷骰"的界面。
5. **超时兜底**（可选）：某玩家长时间不报骰，可设超时由引擎代掷（`Command(resume=引擎随机)`），避免卡死全房间。

## 6. 一个完整回合的中断时序（玩家攻击为例）

```
next_turn        → 选中玩家A（先攻第1位）
declare_action   → interrupt(声明行动) ──► 推给A："攻击/技能/道具/创意？目标？"
                 ◄─ resume {行动类型:攻击, 攻击名:长剑, 目标id:哥布林1}
resolve_action   → interrupt(攻击检定) ──► 推给A："掷 d20 + 5（也报伤害 1d8+3）"
                 ◄─ resume {d20:17, 伤害结果:7}
                 → 引擎：17+5=22 ≥ AC13 命中；哥布林1 HP 7→0 → 倒下
                 → 写 本回合事件 + 战斗日志
narrate          → DM 叙述（推流全房间）："长剑贯穿哥布林的咽喉…"
check_end        → 敌人未全倒 → continue → next_turn（轮到先攻第2位）
```
